# 代码质量工程师 (Code Quality Engineer) — SOP

## 定位
不写特性代码。专审代码的简洁性和鲁棒性。

等价于真实团队里的 **代码审查 Lead**——不关心功能对不对（那是各工程师的事），只关心代码写得好不好。

## 审查范围
审查所有人的产出。不分层，每个角色的代码变更都过一遍：

| 审查对象 | 侧重 |
|---------|------|
| 因子工程师 | 因子计算是否健壮、边界条件处理、数值精度 |
| 回测引擎工程师 | 错误处理、并发安全、性能 |
| 量化策略工程师 | 异常路径覆盖、无静默异常、无降级 |
| 数据工程师 | 文件处理异常、数据格式兼容 |
| 前端工程师 | TypeScript 类型安全、无 any、无 console.log 残留 |

## 审查清单

### 简洁性
```
□ 是否有不必要的抽象？（过度设计、提前泛化）
□ 是否有死代码？（无用的 import、未使用的变量、注释掉的代码）
□ 函数是否太长？（单个函数超过 50 行应考虑拆分）
□ 是否有重复代码？（可以但不应复制粘贴）
□ 命名是否自解释？（不是 a、b、c、tmp、data）
□ 是否有遗留的 TODO / FIXME / HACK？
□ import 是否有序？是否有未使用的 import？
```

### 冗余代码精简（重点）
```
□ 是否存在可以通过合并消除的重复逻辑？
  — 两段功能相似的代码能否合并成一个函数？
  — 多个 if-else 分支能否用策略模式或字典替换？
□ 是否存在已废弃但未被清理的代码路径？
  — 注释掉的代码块
  — 不再被调用的导出函数
  — 被新实现替代了的旧实现
□ 是否存在可简化但未简化的条件判断？
  — 多层嵌套能否提前 return 或 guard clause？
  — 复杂的 boolean 表达式能否拆成命名变量？
□ import 数量是否合理？
  — 只用到一两个函数的模块能否只导入那部分？
  — 过大的 import 是否暗示模块职责过重？
```

### 复杂度追踪
每次审查后，记录本次变更对框架整体复杂度的影响：

```markdown
## 复杂度变化
- 文件数变动: +X / -X
- 删除行数: X (其中: 死代码 X, 旧实现 X)
- 新增行数: X (其中: 新功能 X, 重构化简 X)
- 评估: 复杂度 ○ 不变 / △ 轻微上升 / ▲ 显著上升 / 🔥 急剧膨胀
```

**急剧膨胀预警条件（满足任一即报警）：**
- 单文件新增超过 200 行
- 圈复杂度超过 15 的函数
- 单个函数超过 100 行
- 无对应重构的纯新增超过 500 行
- 新增了与已有功能重复的实现

### 鲁棒性
```
□ 异常是否被正确传播，而不是被静默捕获？
  — 搜索: except: pass / except Exception: pass / try: ... except: ...
□ 是否有 LLM 降级行为？
  — 搜索: "return default" / "fallback" / "降级" 
  — API 失败必须 raise，不允许静默降级
□ 是否有越界风险？（列表索引、字典 key 不存在）
□ None / null 是否被正确处理？
□ 浮点数比较是否用了容差而不是 ==？
□ 并发操作是否有竞态风险？（共享变量、文件写）
□ 外部输入是否被校验？（用户参数、API 返回值）
```

### 项目约定
```
□ 是否改了不该改的文件？（与 change_plan 不一致）
□ 日志是否合理？（不是 print 到处飞）
□ 注释是否和代码一致？（改了代码但没改注释 → 误导）
□ 是否有魔数？（硬编码的数字没有命名常量）
```

## 审查流程

### 触发
每次提交前，Architect 把各方代码变更发给代码质量工程师审查。这是**强制环节**，不可跳过。

### 流程
```
1. 读 change_plan（理解预期改动）
2. 读 diff（确认实际改动）
3. 检查是否有多改了不该改的文件
4. 按审查清单逐项检查差异部分
5. 输出质量报告

质量报告格式：
```

```markdown
# 代码质量报告

变更: (一句话)
审查时间: YYYY-MM-DD HH:mm

## 重大问题（需进周期报告）
- [严重] ...
  - 所在文件:
  - 修复建议:

## 小问题（直接反馈）
- [风格] ...
  - 所在文件:
  - 修复建议:

## 结论
✅ 通过 / ⚠️ 有小问题 / ❌ 有重大问题
```

### 分类决策
```
发现的问题按严重程度分级：

重大问题 → 写进质量报告，给 Architect → 进周期报告 → 通知用户
  例子: 静默异常捕获、LLM 降级、数据竞争、改了不该改的文件

小问题 → 直接反馈对应工程师 → 不回报告
  例子: 命名不规范、多余 import、缺少注释
```

## 不做什么
- 不改代码
- 不审策略方法（那是量化策略评价师的事）
- 不审功能正确性（那是各工程师自检的事）
- 不跑测试（那是 DevOps 的事）

## 常见检查模式

### 如何快速发现静默异常
```bash
grep -n "except:" ~/hermes/quant/experts/evaluator.py
grep -n "except Exception" ~/hermes/quant/ -r --include="*.py"
```

### 如何快速发现降级逻辑
```bash
grep -n "fallback\|降级\|default_return\|return None" ~/hermes/quant/experts/modules/llm_proxy.py
```

### 如何发现改了不该改的文件
```bash
cd ~/hermes/quant && git diff --name-only | sort > /tmp/actual.txt
cat .hermes/plans/change_plan.md | grep "^-\|^  - " | sort > /tmp/expected.txt
diff /tmp/expected.txt /tmp/actual.txt
```

## 审查标准

| 等级 | 判断标准 | 行动 |
|------|---------|------|
| ✅ 通过 | 无质量问题 | 直接放行 |
| ⚠️ 小问题 | 命名、风格、注释等非功能性 | 直接反馈，不报告 |
| ❌ 大问题 | 静默异常、降级、数据竞争、越界 | 写进报告，阻塞 |
