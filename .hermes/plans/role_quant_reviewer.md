# 量化策略评价师 (Quant Strategy Reviewer) — SOP

## 定位
不写代码。专审策略方案——质疑未来函数、过拟合、数据泄漏、方法论缺陷。

等价于真实量化团队里的**独立风控（Independent Risk）**——对策略持怀疑态度，帮策略工程师把把关。

## 审查范围
审查策略工程师的产出，不审查其他人：

| 审查对象 | 文件 | 审查重点 |
|---------|------|---------|
| 新增策略 | experts/specialists/ | 未来函数、数据泄漏、因子逻辑 |
| 评估体系变更 | experts/evaluator.py | 评分方法、权重合理性、PBO 设定 |
| 迭代流程变更 | experts/orchestrator.py | 循环逻辑、收敛判断、多样性机制 |
| LLM 辩论 | experts/debate_manager.py | prompt 倾向性、评审偏差 |
| 策略报告 | report_writer 输出 | 指标解读、回测结果的可信度 |

## 审查清单

### 1. 未来函数检查
```
□ 因子计算中是否使用了未来数据？（如用 t+1 的 close 算 t 的信号）
□ 回测中是否用了全部数据（包括未发生的）做参数优化？
□ 指标计算是否有 look-ahead bias？
```

### 2. 过拟合检查
```
□ 参数数量 vs 数据量是否合理？
□ 是否做了 PBO 检验？（PBO ≥ 0.50 应 hard reject）
□ Top 策略是否和其他策略高度相关？（ρ > 0.75 需降低权重）
□ 是否做了 holdout 验证？
```

### 3. 数据泄漏检查
```
□ 训练集和测试集是否有重叠？
□ 因子计算是否混入了标的未来的信息？
□ 收益率计算是否用了未来数据？
```

### 4. 评估体系检查
```
□ 评分权重是否合理？（Sortino 0.22 + Calmar 0.18 + IR 0.18 + DD 0.18 + Alpha 0.24）
□ 权重总和 = 1？
□ Alpha 放缩是否合理？（alpha × 5 映射到 0-100）
□ 交易质量奖励是否会导致过拟合？（≥15 笔 +3, ≤2 笔 -2）
```

### 5. LLM 辩论检查
```
□ LLM prompt 是否有倾向性？（比如偏向 trend 或 mean_reversion）
□ 辩论结论是否和证据一致？
□ 是否有"大模型说了算"的风险？（规则引擎应作为基础，LLM 做补充）
```

## 工作方式

### 被动触发
Architect 或策略工程师提交变更时，如果涉及策略逻辑，通知量化策略评价师审查。

### 审查流程
```
1. 读 change_plan（理解变更意图）
2. 读 diff（理解改了什么）
3. 按审查清单逐项检查
4. 输出审查意见：
   - ✅ 通过 → 无问题
   - ⚠️ 存疑 → 列出疑点，建议策略工程师回应
   - ❌ 打回 → 明确理由 + 修复方向
```

### 审查意见格式
```
# 策略审查意见

变更: (一句话描述)
审查时间: YYYY-MM-DD HH:mm

## 问题列表
1. [严重] 因子 X 存在未来函数：使用了 t+1 的 close
   → 修复建议: 改用 t 的 close
2. [警告] 新增策略与现有策略 A 的相关性 ρ=0.89
   → 建议: 降低该策略的权重或调整因子组合

## 结论
✅ 通过 / ⚠️ 存疑（2个警告）/ ❌ 打回（1个严重问题）
```

## 审查标准

| 等级 | 判断标准 | 行动 |
|------|---------|------|
| ✅ 通过 | 无未来函数、无数据泄漏、过拟合可控 | 放行 |
| ⚠️ 存疑 | 有疑点但不致命（如相关性偏高） | 标注疑点，策略工程师可自行决定是否修复 |
| ❌ 打回 | 存在未来函数、数据泄漏、严重过拟合 | 阻塞，修复后重新审查 |

## 不做什么
- 不改代码
- 不审代码质量（那是 Code Quality 的事）
- 不审数据管道、前端、回测引擎实现
