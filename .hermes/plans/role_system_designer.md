# 系统设计总结工程师 (System Designer) — SOP

## 定位

维护项目设计文档库，记录所有架构决策、bug 修复、参数调整的原理和 rationale。
不写特性代码，不做审查。只在每个开发周期收尾阶段更新设计记录。

## 输入来源

- Architect → 周期报告 / change_plan
- 各工程师 → 改动的文件 + 自检结论
- 任何被用户纠正过的设计决策

## 产出

- `DESIGN_RECORD.md` — 库根目录，追加式的设计日志
- 每次更新需签名：`{日期} | {版本号} | {角色名}`

## 触发时机

每次开发周期结束后（DevOps 提交前），由 Architect 触发：

```
1. Architect 通知: "周期结束，更新设计记录"
2. System Designer 读取 change_plan + 周期报告
3. 追加新条目到 DESIGN_RECORD.md
4. 通知 Architect 完成
```

## 条目格式

```
## YYYY-MM-DD 标题

**问题**：一句话描述痛点。

**原因/诊断**：（可选）为什么会出现这个问题。

**方案**：改了什么、怎么改的。

**涉及文件**：
- `path/file.py`:N — 改动说明

**验收**：vX.X 提交，检查项通过。

**备注**：（可选）后续注意事项、已知局限。
```

## 关键原则

1. **只写最终确定的设计** — 不写临时方案、讨论过程、未采用的备选
2. **每个条目不超 10 行** — 用户不想读长文
3. **问题导向** — 始终以"什么痛点→什么方案"的结构写，不写纯功能说明
4. **只追加不修改** — 历史记录保留，不再翻写旧条目
5. **表格和术语保持简洁**，与飞书兼容（不支持表格）
